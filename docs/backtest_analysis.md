## Найденные проблемы

### Проблема 1 — Нет фильтра по размеру ORB (критическая)

**Факты из данных:**

| Диапазон ORB | Сделок | Winrate | Суммарный P&L |
|---|---|---|---|
| 0–150 пп | 11 | 36.4% | +106 |
| 150–250 пп | 23 | 34.8% | +266 |
| 250–350 пп | 20 | 40.0% | +229 |
| 350–450 пп | 16 | 50.0% | +1744 |
| **500+ пп** | **5** | **20.0%** | **−1476** |

8 сделок с ORB > 450 пп дают −689 пп суммарного убытка.  
Без них: 67 сделок, P&L = **+1558 пп** (почти в 2 раза лучше).

**Причина:** широкий ORB = большой стоп = огромный убыток при проигрыше.  
Март 2026 — серия из 16 убыточных сделок подряд, многие с ORB 440–688 пп.

**Исправление:** добавить параметр `MAX_ORB = 400` в `strategy.py` / `run_backtest.py`.  
Пропускать дни, где `orb.size > MAX_ORB`.

---

### Проблема 2 — Оптимистичная цена входа в бэктесте

**Код (strategy.py:83–91):**
```python
if close > orb.high and close > current_vwap:
    entry = orb.high  # ← ошибка: цена уже ушла дальше
```

Сигнал формируется когда `close > orb.high`, то есть свеча уже закрылась за пределами ORB.  
Бэктест считает вход по `orb.high`, но реальный рыночный ордер исполнился бы по ~`close` (или открытию следующей свечи).

**Последствие:** реальный риск на сделку в 1.3–1.5 раза выше, чем показывает бэктест.  
P&L бэктеста завышен.

**Исправление:** использовать `entry = close` (цена закрытия сигнальной свечи).  
Это даст реалистичный расчёт стопа и тейка.

---

### Проблема 3 — Баг симуляции: порядок проверки стоп/тейк

**Код (run_backtest.py:59–76):**
```python
if candle["low"] <= sig.stop:      # стоп проверяется ПЕРВЫМ
    return Trade(..., "loss", ...)
if candle["high"] >= sig.take:
    return Trade(..., "win", ...)
```

Если в одной свече `low ≤ stop` **и** `high ≥ take` одновременно — код всегда засчитывает убыток.  
В реальности неизвестно, что было раньше (нужны тиковые данные).

**Исправление:** определять по тому, какой уровень ближе к цене открытия свечи.  
Если `candle["open"]` ближе к стопу — стоп первым, иначе — тейк первым.
